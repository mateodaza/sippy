Title:
Sippy: WhatsApp Stablecoin Payments for Latin America



Project Details:

Sippy enables users to send, receive, and hold USDC stablecoins through WhatsApp messages. No app downloads. No seed phrases. No crypto knowledge required.

WHY "SIPPY":

Sippy is designed to be the first taste of Web3 for users who would never otherwise try it. Like a sippy cup for a child learning to drink, we provide a safe, simple introduction to blockchain - not the full complexity. Users don't need to understand Arbitrum, gas fees, or wallets. They just send money to friends via WhatsApp.

THE PROBLEM WE'RE SOLVING:

Mainstream crypto adoption is blocked by complexity: app downloads, seed phrases, gas fees, and technical jargon. Even "simple" wallets require understanding foreign concepts. Sippy removes ALL of this - users interact with USDC through the app they already use daily (WhatsApp has 90%+ penetration across Latin America).
We're creating the on-ramp for users who would never download a wallet app.

WHAT WE'VE BUILT (Live on Arbitrum One):

- Working WhatsApp bot that processes natural language payment commands
- LLM engine (Groq - Llama 3.3 70B) handling Spanish and English
- Non-custodial embedded wallets via Coinbase CDP
- Spend permission system with user-controlled daily limits
- Gasless transactions (users never pay fees)
- GasRefuel smart contract deployed on Arbitrum One mainnet
- Web UI for wallet setup and settings management (sippy.lat/setup)

CURRENT TRACTION:

- ETHOnline 2025 Finalist
- End-to-end payment flow validated on mainnet
- WhatsApp Business API approved (2,000 bot-initiated messages/day + unlimited user-initiated)
- Fiat on/off ramp partner identified with agreement in place

TECH STACK:

- Wallet Infrastructure: Coinbase CDP Embedded Wallets (non-custodial smart accounts)
- Spend Permissions: SpendPermissionManager contract for user-approved transfers
- Smart Contract: GasRefuel.sol on Arbitrum One (gas sponsorship)
- Messaging: WhatsApp Business API via Meta Cloud API
- Backend: Node.js/TypeScript with Groq LLM (Llama 3.3 70B)
- Database: PostgreSQL for phone-to-wallet registry
- Token: USDC on Arbitrum One
CUSTODY MODEL:

Non-custodial via Coinbase CDP Embedded Wallets. Each user gets their own smart account authenticated via SMS. Users control their funds completely - Sippy operates through spend permissions that users can revoke anytime.

How it works:
1. User creates wallet via sippy.lat/setup (SMS verification through CDP)
2. User grants Sippy a spend permission with their chosen daily limit
3. When user sends via WhatsApp, Sippy's spender wallet executes the transfer using the permission
4. User can revoke permission or change limits anytime via settings

Sippy never holds user funds. The spend permission model means onramp providers can send directly to user wallets with clear regulatory separation.

WHY ARBITRUM (LONG-TERM COMMITMENT):

Arbitrum is our foundational settlement layer for the foreseeable future.

Why Arbitrum:
- Cost: Arbitrum's low fees (~$0.01/tx) are essential for our "near-zero cost" value proposition

AML / KYC & fraud prevention (proportional by design)

• user-controlled daily spend limits (enforced onchain),
• transaction monitoring and reporting where required,
• clear terms, conditions, and policies to prevent fraud and misuse,
• clean separation of responsibility between the protocol and third-party providers.

Why this matters

• Arbitrum gains wallets and users immediately

In short, the DAO is funding a forward-compatible legal foundation that enables safe launch today and optional evolution tomorrow — without risk, rework, or dependency.

ONRAMP STRATEGY:

We've identified a fiat on/off ramp partner with an agreement in place. Our non-custodial architecture makes this integration cleaner:

- Users control their own wallets (CDP Embedded Wallets)
- Onramp sends USDC directly to user's wallet address
- Sippy doesn't touch user funds during onramp
- Clear regulatory separation between Sippy (payments) and Onramp (fiat conversion)

Target flow: User sends COP via local payment method -> Onramp converts -> USDC lands in user's Sippy wallet -> Ready to send via WhatsApp.

This is similar to how PayPal handles third-party cards - the user experience is seamless, but the regulated entity (Onramp) handles fiat conversion.

LLM SAFEGUARDS:

1. Structured Parsing:
LLM extracts intent into structured format (action, amount, recipient) - never executed directly from raw text. If parsing confidence is low, system asks for clarification.

2. Explicit Confirmation:
Before any transfer, user sees:
"Send $25 USDC to +57 300 XXX XXXX (Maria)? Reply YES to confirm."

3. Amount Sanity Checks:
- Transactions > user's daily limit: Blocked (enforced on-chain via spend permissions)
- Unusual amounts flagged for confirmation

4. Recipient Validation:
- System confirms recipient exists before execution
- New recipients flagged: "This is a new contact. Please verify the number."
- Phone number format validation

5. User-Controlled Settings:
Users can customize via settings page:
- Daily spend limit (enforced on-chain)
- Confirmation preferences
- Phone visibility
